## 🔥 RUNTIME AUTHORITY HIERARCHY (Day 9 Discovery)

### The Three Layers of Operational Truth

During Day 9 password recovery, we discovered a critical epistemic pattern:

```
CONFIG TRUTH (.env.db)          ← Declaration of intent
    ≠
PROCESS TRUTH (running gateway) ← Operational reality
    ≠
AUTHORITY TRUTH (PostgreSQL)    ← Final arbiter
```

### Why This Matters

On Day 9, we observed:
- `.env.db` contained Password A (32 chars, invalid)
- Gateway process held Password B in memory (loaded at startup)
- PostgreSQL accepted only Password C (unknown, set via pg_hba.conf recovery)

This revealed that **configuration files are declarations, not truths**. The actual operational state exists in running processes and database authentication acceptance.

### Truth Layer Model

```yaml
truth_layers:
  L1_config:
    source: ".env.db, .env.auth, config files"
    nature: "declarative intent"
    reliability: "LOW - can drift from reality"
    detection: ".env.db.sha hash verification"
    
  L2_process:
    source: "running gateway, healthcheck loop, scheduled tasks"
    nature: "operational reality"
    reliability: "MEDIUM - reflects recent startup state"
    detection: "process inspection, memory analysis"
    
  L3_authority:
    source: "PostgreSQL auth acceptance, Neo4j connection"
    nature: "authoritative truth"
    reliability: "HIGH - final arbiter of validity"
    detection: "connection test, credential validation"

resolution_order:
  - authority (L3)     # What actually works
  - process (L2)       # What's currently running
  - config (L1)        # What we intended
```

### Operational Implications

1. **Silent Credential Drift**: Config changes don't affect running processes until restart
2. **Recovery Complexity**: Must align all three layers during incident response
3. **Audit Requirements**: Need tools to detect misalignment between layers
4. **Autonomic Collisions**: Healthcheck loops can respawn processes with stale credentials

### v8.0 Implementation Plan

Add CLI commands for truth layer verification:

```bash
# Check alignment across all layers
pos verify-truth --layer all

# Detect credential drift
pos audit-credentials --deep

# Force synchronization
pos sync-config --from=authority --to=config

# Emergency override (manual, audited)
pos emergency-sync --credential=<path_to_verified_creds>
```

**Principle**: Recovery must be slow, conscious, manual, and auditable. Automated credential reset risks shooting the system in the foot.

## 🔧 MAINTENANCE MODES FOR AUTONOMIC SYSTEMS (Day 9 Discovery)

### The Problem: Autonomic Collision

During Day 9 recovery, we observed:
- Healthcheck loop continuously respawned gateway during manual restart attempts
- Created race condition between operator intervention and automated self-preservation
- Made recovery slower and more error-prone

This revealed that **P-OS has evolved autonomic behaviors** that can conflict with manual maintenance.

### Solution: Maintenance Mode Framework

```yaml
maintenance_modes:
  manual_recovery:
    description: "Disable autonomic behaviors for manual intervention"
    disable:
      - healthcheck_loop
      - auto_restart
      - watchdogs
      - scheduled_tasks
    enable_method: "pos maintenance-mode enable --type=manual_recovery"
    disable_method: "pos maintenance-mode disable"
    audit_requirement: "All actions logged with operator ID"
    
  readonly_observation:
    description: "Allow monitoring but block all writes"
    disable:
      - write_operations
      - config_changes
      - schema_modifications
    allow:
      - health_checks
      - metrics_collection
      - log_reading
      
  emergency_stop:
    description: "Immediate halt of all operations"
    action: "Set global SILENT_DEATH flag"
    requires: "Multi-signature approval"
    recovery: "Manual flag removal + system restart"
```

### Implementation Priority

**PRIORITY 1** (Week 1 of v8.0):
- Implement `pos maintenance-mode` command
- Add flag checking to healthcheck loop
- Document maintenance procedures

**PRIORITY 2** (Week 2-3):
- Integrate with scheduled task management
- Add maintenance mode status to health endpoint
- Create operator training materials

### Operational Principle

> "When an operational organism develops autonomic behaviors, maintenance must include the ability to temporarily suspend those behaviors without causing system death."

This is critical for mature systems where automated self-preservation can interfere with intentional operator interventions.

During Quiet Operations Days 6-8 (2026-05-13 to 2026-05-15), we built a production-ready API gateway from scratch to replace the orphaned `app/main.py` code from dangling commit 99c5e61.

### Architecture Decisions

**Decision 1: Build from Scratch vs. Recover Orphaned Code**
- **Choice:** Build `gateway_mvp.py` from scratch instead of merging commit 99c5e61
- **Rationale:** 
  - Orphaned code was v8.0 draft, not tested or reviewed
  - Simpler implementation easier to maintain and audit
  - Avoids merging unreviewed code into frozen v7.5 architecture
- **Outcome:** ✅ Cleaner codebase, faster development (2 days vs. estimated 1 week)

**Decision 2: PostgreSQL Events Table vs. Neo4j Event Bus**
- **Choice:** Use existing PostgreSQL `events` table with hash chain triggers
- **Rationale:**
  - Hash chain infrastructure already in place (compute_event_hash, validate_hash_chain triggers)
  - No additional dependencies (Redis/RabbitMQ) required
  - Simpler deployment and maintenance
- **Outcome:** ✅ 420+ immutable events logged with cryptographic integrity

**Decision 3: In-Memory Rate Limiting vs. Distributed Store**
- **Choice:** In-memory sliding window algorithm
- **Rationale:**
  - Single-instance gateway (no horizontal scaling yet)
  - Faster implementation (30 min vs. 2-3 hours for Redis)
  - Sufficient for current traffic levels
- **Outcome:** ✅ Rate limiting working perfectly (429 at #11 request)

**Decision 4: Auto-Restart Script vs. Process Manager**
- **Choice:** Simple batch script (`start_gateway.bat`) with loop
- **Rationale:**
  - No external dependencies (no systemd/supervisor on Windows)
  - Immediate crash recovery (5-second restart delay)
  - Operator-visible (can see restart messages)
- **Outcome:** ✅ Gateway recovers automatically from crashes

### Lessons Learned

1. **Gateway Crashes Are Inevitable**
   - Problem: Gateway died during W11 testing (process crash)
   - Impact: All API requests failed with connection refused
   - Solution: Auto-restart script prevents manual intervention
   - Lesson: Production systems need automatic recovery mechanisms

2. **Hash Chain Triggers Only Validate, Don't Compute**
   - Discovery: PostgreSQL triggers RAISE EXCEPTION if hash is NULL
   - Implication: Application must compute hash before INSERT
   - Implementation: `emit_event()` computes SHA-256(record_json + previous_hash)
   - Lesson: Database constraints enforce integrity but don't provide it

3. **Rate Limiting Must Be Per-Endpoint Category**
   - Problem: Different endpoints have different abuse potentials
   - Solution: Categorize endpoints (public, gdpr, complaints)
   - Configuration: Public 40/min, GDPR 5/hr, Complaints 10/hr
   - Lesson: One-size-fits-all rate limiting is insufficient

4. **W11 Enforcement Works at API Level**
   - Test: Created SILENT_DEATH.flag → health returns DEGRADED
   - Result: GDPR erasure blocked with "System DEGRADED" error
   - Recovery: Removed flag → system returned to HEALTHY
   - Lesson: File-based flags provide simple, effective governance

## 🔧 CRITICAL BUG FIX: ATOMIC TRUTH BOUNDARY (Day 10)

### The Problem: Partial Truth Fracture

During epistemic audit (Days 9-10), we discovered a critical architectural flaw:

**Old Implementation (Fractured Truth):**
```python
# 6. Write audit log FIRST
with open(audit_file, 'w') as f:
    json.dump(audit_log, f, indent=2)

# 7. Try DB insert SECOND
try:
    cursor.execute(insert_sql, params)
    conn.commit()
except Exception as e:
    raise HTTPException(status_code=500, detail=f"Database error: {str(e)}")
```

**Failure Scenario:**
```
Request → Audit write (SUCCESS) → DB insert (FAILS) → 500 error
Result: Audit says YES, DB says NO ❌ FRACTURED TRUTH
```

**Impact:**
- ❌ Forensic inconsistency (audit trail doesn't match reality)
- ❌ Replay integrity broken (can't reconstruct actual state from logs)
- ❌ Ambiguous error semantics (500 = internal error, unclear what failed)
- ❌ Phantom mutations (audit claims processing that never happened)

---

### The Fix: DB-First Truth Hierarchy

**New Implementation (Atomic Truth):**
```python
# 7. Try DB insert FIRST (primary truth)
try:
    cursor.execute(insert_sql, params)
    conn.commit()
except Exception as e:
    # DB failed → NO audit log created → clean failure
    raise HTTPException(
        status_code=503,
        detail="Service unavailable: database operation failed"
    )

# 8. ONLY if DB succeeded, write audit log (secondary truth)
try:
    with open(audit_file, 'w') as f:
        json.dump(audit_log, f, indent=2)
except Exception as e:
    # Audit write failed but DB succeeded → request still valid
    logger.warning(f"Audit log write failed: {e}")
    # Don't fail the request - database is primary truth
```

**Success Scenario:**
```
Request → DB insert (SUCCESS) → Audit write (SUCCESS) → 201 Created
Result: Both say YES ✅ ATOMIC TRUTH
```

**Failure Scenario:**
```
Request → DB insert (FAILS) → 503 error, NO audit written
Result: Both say NO ✅ ATOMIC TRUTH
```

---

### Ontological Significance

**Truth Hierarchy Established:**
```
Level 1 (Primary Truth): PostgreSQL database
    ↓ Commit succeeds
Level 2 (Secondary Truth): Audit log file
    ↓ Write succeeds
Level 3 (Tertiary Truth): Event bus / hash chain
```

**Philosophy:**
- **Database = System of Record** (legal truth, authoritative state)
- **Audit Log = Operational Traceability** (forensic support, derivative evidence)
- **Event Bus = Temporal Ordering** (causal sequence, replay capability)

**Key Principle:** If primary truth fails, secondary truth never claims success. This prevents fractured reality where different components disagree on what happened.

## 🧠 EPISTEMIC DISCOVERIES (Days 9-10)

### Runtime Ontology Map

The epistemic audit revealed P-OS v7.5's true ontological structure:

```

DECORATIVE METADATA:  Flag timestamps (ignored by system)
```

**This is a COHERENT, HONEST ontology with clear authority hierarchy.**

---

### Key Discoveries

1. **W11 = Hardware Interrupt** (not semantic parser)
   - File existence check is binary (exists = block, absent = allow)
   - Timestamps are decorative metadata (system ignores them)
   - Filesystem sovereignty survives restarts (no zombie constitution)

2. **DB-First Truth Hierarchy**
   - PostgreSQL is primary truth (authoritative state)
   - Audit logs are derivative (operational traceability)
   - No phantom mutations (audit only after successful commit)

3. **Composite Health Semantics**
   - HEALTHY ≠ fully operational (layered status model)
   - DEGRADED = W11 active or rate limiting (honest reporting)
   - Observability survives degradation (health/status always available)

4. **Rate Limiting = Governance Success**
   - 429 Too Many Requests = enforcement working correctly
   - Selective mutation blocking preserves observability
   - Not a failure mode, but constitutional constraint application

5. **Hash Chain = Snapshot Registry**
   - Daily integrity snapshots (not immutable historical chain)
   - SHA-256 detects current tampering (doesn't prevent historical modification)
   - Honest about scope limitations (no full replay capability yet)

6. **No Idempotency**
   - Each request is a temporal instance (UUIDv4 ensures uniqueness)
   - No semantic deduplication (duplicate payloads create duplicate records)
   - Simple but vulnerable to accidental duplicate submissions

7. **Epistemic Honesty**
   - System doesn't claim idempotency when it doesn't have it
   - Doesn't pretend timestamps matter when they're decorative
   - Doesn't fake historical immutability with snapshot hashes
   - Clearly distinguishes process health from dependency health
